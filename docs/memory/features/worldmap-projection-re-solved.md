**The answer:** the native map projects EVERY point via a live converter array inline in `CS::WorldMapViewModel` (vtable RVA `0x2ad82e0`): `converters = VM+0xF8` (stride 0x30, 8 slots), `count = VM+0x280` (=8). Per-point projection = `FUN_140876140` (RVA `0x876140`), verified math: `worldX = posX + (gridX − c.gridXbase[+0xA])·256; mapX = (worldX − originX[+0xC])·scale[+0x20] + biasX[+0x18]; mapZ = −(worldZ − originZ[+0x14])·scale + biasZ[+0x1C]`. Converter `+0x28` = WorldMapLegacyConvParamGroup node → `FUN_1408775e0` (RVA `0x8775e0`) folds legacy dungeons LIVE (RB-tree, translates pos by conv base-offset — the proper fix the mod's `project_dungeon_row_to_overworld` only approximated = area-16 wrong-region bug). Overworld slot = scale 1, originX 7168, originZ 16384, bias 128/128 → reduces to our baked −7040/+16512. Page = matched-slot index into `(&DAT_142ad82f8)` = bytes `[00 01 0a]` = {0 overworld, 1 underground, 10 DLC}, + area==12 ⇒ underground override (`FUN_140887870`). Callable wrapper = `FUN_1408877d0(VM,&outMapXZ,&packedId,&worldLocal)` with `packedId=(area<<24)|(gridX<<16)|(gridZ<<8)`, `worldLocal={posX,posY,posZ}` (AREA-LOCAL, NOT grid*256+pos — same frame as player-pos +0x6C0). m19/Chapel no-conv areas → no converter accepts → game DOESN'T place them → gate.
