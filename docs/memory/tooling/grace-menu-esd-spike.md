Spike 2026-06-18 to judge putting a MapForGoblins settings option INTO ERR's site-of-grace menu (user wants "menu dans la grace comme ERR"). See [[thread3-grace-toggle]] (why we went DLL-native F7/F8) and [[mapforgoblins-map-open-freeze]].

**KEY: ERR's grace settings menu = talk-ESD, NOT the DLL.** reforged.dll (Rust) feature list = all_knowing_sign/chr_data_store/discord/equip_load/gui_overlay(debug_gui)/hooks/misc/on_init/sanity_check/stutter_fix/waygate — NO menu/talk/grace feature. The menu lives in `<ERR_ROOT>/mod/script/talk/m00_00_00_00.talkesdbnd.dcx` (524KB, DFLT-compressed BND4 of 14 .esd EzState machines) + FMG text. Same system the original Map for Goblins (Nexus 3091) used; ERR overwrote the file → collision is why MFG is a DLL.

**Toolchain (works on Linux, no Windows needed):** `tools/esd_dump/` (C#, net10.0, dotnet 10). References ONLY `tools/lib/Andre.SoulsFormats.dll` (Smithbox/DSMapStudio fork — HAS SoulsFormats.ESD) + the other lib dlls; Andre.IO/Andre.Core NOT needed, SMITHBOX_DIR unset. `DCX.Decompress` returns `Memory<byte>` (call `.ToArray()`). Target is **DFLT (zlib), NOT Oodle** → oo2core not required. ESD API: `ESD.Read(byte[])`; `StateGroups` = Dictionary<long,Dictionary<long,State>>; State.{Entry,While,Exit}Commands (List<CommandCall>) + Conditions; CommandCall.{CommandBank,CommandID,Arguments(List<byte[]>)}; Condition.{TargetState,Evaluator(byte[]),PassCommands,Subconditions}. ESD : SoulsFile<ESD> → Write() inherited (round-trip viable). **NO EzSemble** bytecode assembler in the dll → authoring new menu entries = hand-writing EzState arg bytecode. Modes: `--hist` (per-esd bank:id histogram), `--menu <id>` (states calling talk cmd 1:id), `--full`. esd_dump's arg decoder is BROKEN for textIds (prints denormal floats) — only structure is reliable, not labels.

**Menu mechanism decoded (bank 1 = talk commands):** AddTalkListData = **1:19** (adds a selectable line: index, textId, …). ClearTalkListData/open = **1:20**. Show/request = **1:10**. Selection routes via Condition.Evaluator (GetTalkListEntryResult comparison bytecode) → TargetState. Bank 5 = text/SetTalkListEntry-with-FMG-ref (5:19, 5:149). Bank 6:big-int = ESD function calls. Bank 7:-1 = getter. Menus live in t000001000 (355 menu states), t000001300 (675!), t000001010/t000004000 (140 each — these two are byte-identical dupes). Esd files: t000001000(1.4MB)/t000001010/t000004000(1MB dupes)/t000001300(462KB) are the big dialogue trees.

**Feasibility verdict: ESD-merge is technically GO but strategically POOR.** Blockers: (1) ERR OWNS + re-ships the 524KB talkesdbnd EVERY version → any merge is a build-time patch that breaks on each ERR update; (2) no assembler → hand-author EzState bytecode for the new AddTalkListData line + condition branch; (3) trees are huge (4454/1350 states) → splicing into the right state without breaking ERR flow is delicate. NOT yet confirmed: whether a menu option sets an EVENT FLAG natively (decoder too broken to trace) — load-bearing, because if options set flags the DLL just READS them (no flag-writer RE; original MFG did exactly grace-menu→flip-eventFlag→WorldMapPointParam.eventFlagId gate, so the pattern exists in ER).

**Recommended form if pursued:** an AUTOMATED build-time ESD patcher in tools/ (reads ERR's CURRENT talkesdbnd, injects one option that sets one event flag, writes back) so re-merge per ERR update is a script, not hand-work — needs a tiny EzState assembler for the few opcodes. Cleaner alternatives that avoid touching ERR's file: Route B = DLL hook of the talk-list API (AddTalkListData/CSFD4ScriptManager, no example in reforged.dll, RE-heavy, same wall as live-refresh) OR own ImGui+DX12 overlay (reforged.dll's gui_overlay::debug_gui proves a self-owned overlay runs STABLE in this ERR/Proton stack — EROverlay crash was bad Present sync, not Proton; user disabled EROverlay).
