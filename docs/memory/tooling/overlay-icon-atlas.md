The ImGui overlay map ([[overlay-rendered-markers]]) draws ONE icon per Category from a baked atlas; markers without a cell fall back to a coloured circle.

**Blue/coloured dots are BY DESIGN, not a bug/regression.** 6 categories have `nullptr` icon_key in `src/worldmap/category_meta.cpp` (CAT[]) → circle fallback in the group colour: **EquipTalismans, KeyWhetblades, LootUtilities, LootThrowables, WorldMaps, WorldQuestNPC** (e.g. "Marika's Scarseal" = talisman = blue dot). Documented at category_meta.cpp:11-12. `category_color()` gives the fallback colour by group (Equip=blue, Loot=green, …). To give one a real icon: assign a good iconId (KEY_ICON_OVERRIDE in the generator) + set its CAT[] key non-null + regenerate.

**Atlas storage = PNG-embedded (commit 5db8995, 2026-06-21; ✅ VERIFIED IN-GAME 2026-06-22 #5 — renders perfectly, runtime stb-decode path confirmed).** Was a raw `ATLAS_RGBA[]` byte array (~20.9k-line generated .cpp). Now `ATLAS_PNG[]` + `ATLAS_PNG_LEN` (compressed PNG, ~135 KB) in `src/generated_shared/goblin_overlay_icons.cpp` (~4.3k lines), decoded once at load by `try_upload_atlas` (goblin_overlay.cpp) via `stbi_load_from_memory` → RGBA → DX12 upload (SRV 1) → free. stb_image.h vendored at `third_party/stb_image.h` (v2.30); impl TU = `src/stb_image_impl.cpp` (PNG-only, no stdio). Dropped the dead LOGO_RGBA (no consumer). ATLAS_W/H/CELL + ICON_CELLS (key→col/row) unchanged → `icon_uv` (map_renderer.cpp) still computes UVs the same. **A runtime for-loop CANNOT generate these — they're rendered ER art, not procedural; PNG-embed is the clean way to shrink the file while keeping visuals.**

**Regenerating the atlas needs FFDEC (NOT available on this Linux box by default).** Pipeline: `tools/generate_overlay_icons.py` derives key→category→iconId from source, renders missing iconIds from the vanilla Scaleform gfx (`assets/menu/02_120_worldmap_vanilla.gfx`) via `tools/render_map_icons.py` → FFDEC (`ffdec-cli.jar`, set `$FFDEC_CLI`; Java 21 present, jar absent → download), composites a 40px-cell atlas, emits the PNG-blob .cpp. `assets/map_icons/by_iconId/` is EMPTY in the repo (the .cpp is the committed output) → a regen re-renders everything. The generator was updated (5db8995) to emit the PNG form; pixels for that commit were re-encoded from the existing RGBA (no FFDEC needed for the format change).
