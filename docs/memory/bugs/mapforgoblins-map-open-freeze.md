**UNLOCK-ALL FLAG-WRITER = DONE (2026-06-20, commit a19f2bc).** `goblin::markers::set_event_flag(id,value)` writes live event flags via EventFlagMan (reader's singleton). SetEventFlag/"EventFlag_C1" resolved by its STABLE PROLOGUE AOB `48 89 5C 24 08 48 89 74 24 18 57 48 83 EC 30 48 8B DA 41 0F B6 F8` (up to the r8b value-capture) → runtime **0x1405d2240**, read-back confirmed flags set. The full Hexinton 30-byte signature does NOT resolve (its tail `8B 12 48 8B F1 85 D2 0F 84…` drifts per game version). Foundation for unlock-all. (Note: revealing the DLC MAP this way FAILED — map reveal is per-tile `WorldMapPieceParam.openEventFlagId`, ERR-remapped, and setting them didn't visually refresh; pragmatic path was a 100% save swap, see [[err-save-file-format]].)

The ~6 s freeze on opening the in-game world map (and on-close micro-freezes) in MapForGoblins (<repo>) is **entirely game-side, not the mod**. Diagnosed 2026-06-17 via `[BENCH]` timers (`src/goblin_bench.hpp`, commit fc95786). Full writeup: `docs/map_open_freeze.md`. See [[mapforgoblins-map-freeze]].

**Mechanism.** The mod does all its per-row work ONCE at init (`setup_mod`, dllmain.cpp:124-133, after `load_delay`), installs NO runtime hooks (`enable_hooks()` queue is empty), and injection is a one-shot **pointer swap** of `WorldMapPointParam`'s backing file (`goblin_inject.cpp:443-449`, `set_param_injection_active` :981) — vanilla ~740 rows → expanded ~9692 rows, resident permanently. Mod init total ~20-31 ms; refresh tick ~0.04 ms. So the mod runs nothing on map open. **The freeze is the GAME re-walking every resident `WorldMapPointParam` row to build the map UI on each open.** Cost tracks row COUNT, not icons drawn: `require_map_fragments=true` still freezes (logs "0 hidden at inject" — fragment/zoom gating hides at draw time but rows stay in the table). Display gating does NOT help.

**Curve (measured), superlinear:** 740 rows instant · 1031 instant · 7635 ~3 s · 9692 ~6 s. First ~6600 injected ≈0.45 ms/row, next ~2000 ≈1.46 ms/row (likely O(n log n) sort/index or cache pressure). → trimming the densest categories pays disproportionately (dropping bulk loot, −2057 rows, already halved 6 s→3 s).

**Fix direction = reduce resident row count** (matches the authors' "general icons" version + community Smithbox trimming). Plan: default the densest low-value loot categories OFF in `goblin_config_schema.cpp` (Crafting Materials, Golden Runes+Low, Smithing Stones Low, Consumables, Gloveworts+Great, Greases, Throwables, Ammo, Stat Boosts, Material Nodes, Utilities), keep navigation/high-value on; document the perf/coverage trade-off. Sub-~1 s budget pinned: **5711 resident rows ≈ 1 s** (the ~12-dense-loot-categories-off set: CraftingMaterials,Consumables,GoldenRunes,Gloveworts,SmithingStones,Throwables,Greases,Ammo,StatBoosts,MaterialNodes,Utilities). That set = candidate default, not yet patched into `goblin_config_schema.cpp`.

**Refined model (A/B tested 2026-06-17):** the open cost = rows of the **currently-displayed map PAGE/area**, NOT the whole table. Confirmed: pan/zoom is smooth (build-once-on-open, not per-frame) AND dropping baked area-61 rows did NOT speed the BASE map → **the game partitions by area/page** (area-61 rows free while viewing base, and vice versa). Within the displayed page, ALL its rows are processed regardless of fragment/collected/zoom visibility (all draw-time gating — rows stay resident). So to cut the cost you must **evict rows from injection**, not gate them.

**Levers (re-evaluated):**
- ❌ **Area-split** (inject only current area 60/61) = DEAD — the game already partitions by page, no gain either way.
- ✅ **Trim categories** — cuts area-60 rows; proven by the curve; loses content.
- ✅ **Fragment-lazy** — don't inject rows for undiscovered map-fragment sub-regions of area 60; re-inject on fragment-flag gain. Early/mid-game win, keeps content (matches the map only showing discovered regions). The mod already knows each row's fragment gate (apply_map_logic sets eventFlagId = the fragment flag) and already polls flags in the refresh loop.
- ✅ **Collected-eviction** — evict collected/killed rows (mod already tracks them to hide); late-game win.
The structural winners are fragment-lazy + collected-eviction (both cut the area-60 resident count without losing content); they stack with trim.

**KEY MECHANISM (A/B confirmed 2026-06-17): `areaNo = 99` evicts a row from the open cost WITHOUT removing it from the table.** Test: set every area-60 non-boss row's areaNo to 99 (resident but off-page) → base map opened INSTANT (vs 6s). So the per-page build skips off-page rows; eviction = a **1-byte toggle (areaNo 60↔99), no table rebuild, no realloc, no 16-align risk.** This IS the existing `hide_icon` trick (goblin_collected.cpp:618-624, `p[0x20]=99`), and `register_param_ptr` already saves the original areaNo + `refresh()` restores/re-hides. Consequences: (1) **collected-eviction already works** — collected rows are set to 99 by hide_icon → already off the cost, so a fully-looted late save is already fast; the pain is early/mid game with lots of uncollected *visible* loot. (2) **fragment-lazy = trivial**: today `require_map_fragments` gates via `eventFlagId` (row stays area-60 → still costs); switch it to set areaNo=99 for undiscovered-fragment rows (restore to 60 on discovery in the refresh loop) → `require_map_fragments=true` becomes FAST, same visuals. (3) **"see ALL markers (all regions, uncollected) + fast" is physically impossible** — cost is per-displayed-row, showing ~9000 = ~6s; must gate something (fragment = the natural gate). Implementation path: make require_map_fragments evict via areaNo (integrate with the collected refresh restore loop so they don't fight).

**Region-lazy by VIEWPORT/zoom = RULED OUT:** build-once-on-open (pan is smooth) → changing the table after open does nothing, and there's no viewport/camera reader (would need new AOB). Region-lazy by **player position** also breaks panning to far/undiscovered regions. The viable "region" axis is map-FRAGMENT (above), not viewport.
