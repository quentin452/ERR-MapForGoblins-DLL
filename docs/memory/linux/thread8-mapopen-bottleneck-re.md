**WHAT WE KNOW (from [[mapforgoblins-map-open-freeze]], measured via [BENCH] timers):**
- Freeze is 100% GAME-side. The mod does its work once at init (one-shot pointer-swap of WorldMapPointParam to ~9692 rows, resident permanently); refresh tick ~0.04ms; NOTHING runs on map open.
- The game re-walks every resident WorldMapPointParam row of the CURRENTLY-DISPLAYED page/area to build the map UI, on EACH open. Cost tracks ROW COUNT, not icons drawn (fragment/zoom/collected gating is draw-time only → rows stay resident → still cost).
- **Curve = SUPERLINEAR:** 740 rows instant · 1031 instant · 7635 ~3s · 9692 ~6s. First ~6600 ≈0.45 ms/row, next ~2000 ≈1.46 ms/row. → smells O(n log n) sort/index, or an O(n²) overlap/dedup/spatial pass, or per-row allocation + cache pressure. THE actual hot routine is unidentified.
- Game partitions by area/page (area-61 rows free while viewing base, and vice versa) — so the bottleneck runs over ONE page's rows at a time, triggered on open (CSMenuMan+0xCD flips to 7).

**WHY RE IT (the payoff):** "see ALL markers (all regions, uncollected) + fast" is currently PHYSICALLY IMPOSSIBLE under the eviction model (cost is per-displayed-row, ~9000 rows = ~6s, must gate something). IF the hot function is a cheapenable algorithm (e.g. a comparison sort we could pre-sort for, an O(n²) we could short-circuit, a redundant per-open rebuild we could cache across opens), a hook/patch could break that ceiling. Exploratory — to learn IF the algorithm can be made cheaper, not a committed feature.
